## 계정 시스템 전체 현황

### 인증 아키텍처
- SyOps가 유일한 인증 서버. 모든 로그인/토큰 발급은 SyOps에서 처리
- JWT Access Token (30분) + Refresh Token (30일, httpOnly Cookie)
- 도메인 쿠키 (`syops_token`, `domain=.syworkspace.cloud`)로 서브도메인 서비스에 자동 전달
- 각 서비스는 `SYOPS_SECRET_KEY`로 JWT를 자체 검증 (SyOps API 호출 없음)

### JWT payload 구조
```json
{
  "sub": "user_id",
  "role": "admin|user",
  "username": "...",
  "services": ["quickdrop", "voca_drill"],
  "exp": ...,
  "type": "access"
}
```

### 권한 3단계
1. **인증**: JWT 서명 검증 (유효한 사용자인가?)
2. **서비스 접근**: `services` 필드 체크 (이 서비스를 쓸 수 있는가?)
3. **역할**: `role` 체크 (admin 전용 기능인가?)
- admin은 모든 서비스 접근 가능 (services 무시)

### 서비스별 적용 현황
| 서비스 | 인증 | 서비스 권한 | 데이터 격리 | 상태 |
|--------|------|------------|------------|------|
| SyOps | Bearer 헤더 | - | - | 완료 |
| QuickDrop | 도메인 쿠키 | quickdrop | user_id별 디렉토리 | 완료 |
| Voca_Drill | - | AVAILABLE_SERVICES에 등록만 | - | 미적용 |

### 대시보드 관리 기능
- 사용자 목록 조회 / 생성 / 비활성화 / 비밀번호 초기화
- 서비스별 접근 권한 토글 (칩 UI)
- 모두 admin 전용 (`require_admin`)

## 핵심 결정 & 맥락
- Lv.1(세션) 건너뛰고 Lv.2(JWT) 직행 — FastAPI 생태계 호환, 구현 난이도 차이 미미
- passlib 호환 문제로 bcrypt 직접 사용
- 서비스 권한은 JSON 컬럼으로 단순 구현 (2-3명 사용자, 소수 서비스에 적합)
- QuickDrop auth_check에서 서비스 권한을 분리하여 무한 리다이렉트 방지
